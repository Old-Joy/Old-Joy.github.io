P7如何快速debug

现在，有很多同学苦于P7冗长而繁琐的debug，笔者在第一次P7上机前一周就写完了P7，并且成功在半天时间内找到所有bug，顺利通过课上强测，因此来讨论区分享分享自己的debug经验。

通过阅读本文，您大致可以学到：如何巧妙构造容易找到问题的测试数据；如何妥善利用课上课下课程组给出的报错；如何理解中断行为的处理等知识。

实现P7的过程中一些值得注意的细节

本来是想完整的讲一遍P7要我们干什么，但是这样好像就是把教程复述了一遍，虽然那个教程本身也并不是很清晰。限于篇幅，这里讲一些笔者认为的在实现P7过程中一些小细节，

外设

根据教程的要求，外设一共由计时器0、计时器1、中断发生器和系统桥组成。计时器是如何实现的，在CPU的角度看来其实不重要，CPU只需要与系统桥交互，获得信息，在需要的时候向系统桥传递信息即可，这也符合了高内聚低耦合的思想。

计时器

这里带大家简要分析一下有关计时器的设计文档以及源码，方便大家理解。

教程里说了，计时器的主要功能，就是根据设定的时间来定时产生中断信号。参考计时器的设计文档和源码会发现，计时器的本质是一个状态机。其内部的部分信号值都是可以通过 store 类指令访问修改的，比如倒计时的初值、计时器的使能、计时器的工作模式等等。

这里结合设计文档，将计时器源码注解了一下，希望可以帮到大家。

中断发生器

中断发生器的实现在testbench内。这边也放一下关于中断部分的源码。

可以发现，当宏观PC走到了预先设定好的目标值后，testbench会直接让中断信号置高，也就是模拟键盘鼠标等给出的中断信号。响应也十分简单，如果这里有一个 store 类指令操作的地址是 32'h7f20 那么相应的中断信号会变回低电平。

注意！ store 类指令在判断操作地址是否为 31'h7f20 时，实现与了一个 32'hfffffffc ，其效果等价于 {m_int_addr[31:2], 2'd0} ，也就是说，你的操作地址后两位并不重要。在很多人的设计中，m_int_byteen 与 m_data_byteen 使能信号的开关与操作地址的判断有关，因此需要注意，并不是操作地址完全等于 32'h7f20 时，m_int_byteen 的使能信号才会打开！ 需要囊括从 32'h7f20 到 32'h7f23 。我有一次提交就卡在这里了

另外，评测机内部的testbench也可能和这个不完全一样，其内部的 count 一定不仅仅局限于1，报错信息中的count可以体现您的微系统响应外部中断的次数是否正确。

系统桥

系统桥是CPU与外设、IM、DM交流信息的渠道。其主要功能，是通过CPU给出对不同地址的访问请求，交给CPU正确的信息，对外设、IM、DM中的寄存器进行正确的读写操作。

说的更详细些，举个例子。在官方给出的计时器源码中，传入的地址 Addr 只有 Addr[3:2] 是有效内容。因为针对每一个计时器，只有3个寄存器是对程序员可见的，对于计时器本身，其只需要明确这3个寄存器的地址即可。而关键的读写哪一个计时器，其相应的使能信号由系统桥判断并给出。

说的更直白些，系统桥负责把CPU给出的访问请求“翻译”一下，并执行相应操作或告知CPU正确的信息。

关于异常

CP0协处理器

我们只需要实现 CP0 中的 SR、Cause 和 EPC 三个寄存器，而仔细阅读了《See MIPS Run Linux》的同学可以注意到 CP0 中其实还有大量的其他寄存器，他们都有不同的、更进阶的功能，这些部件目前我们不需要实现。

另外对于部分 CP0 寄存器，我们实现的内容中只需要其部分域，但是这并不意味着我们在开寄存器的时候，只开这些域就足够了！我们来看《See MIPS Run Linux》中是怎么说的。

说到这里有必要对保留域说明一下。许多未用的控制寄存器域被标记为 “0” 。这些域里的位保证读出为零，写入也没有什么害处（当然写入的值被忽略）。另一些保留域被标记为 “reserved” 或者 “x” ；这时候就要小心了，这些域永远应当写入零或者写入前面读出的值，但不应当认为读出的值一定是零或者其他具体值。

事实上，SR Cause等寄存器的其他域有其他更进阶的作用，不保证课上的测试点不会通过 mtc0 来修改，如果少开了保留域，最终可能会导致 mfc0 的结果错误。

  • 在有关异常debug的过程中，CP0 中寄存器的值是你最好的伙伴，你一定需要观察这些信号然后找到自己的问题所在。这个我们马上讲。

另外，《See MIPS Run Linux》中明确指出了如下内容

k0和k1（通用寄存器 $26-27 ），这两个（由软件约定）预留下来的用于异常处理代码中的通用寄存器。预留至少一个通用寄存器是必要的；而具体预留哪一个则是随意地。
你至少需要用k0和k1中一个来指向为异常处理程序预留的一些内存空间。然后就可以保存东西，必要时还可以用另一个来从控制寄存器倒腾数据。

这也就告诉我们，k0和k1两个寄存器需要被预留下来，因为在异常处理代码中经常会被用到。

Debug time！

终于到debug时间了！
现在，面对着课上冰冷的评测姬，我们要如何获得她的芳心呢？

关注评测机给出的数据

今年，无论是课上课下，我们都能看到评测机给出的报错信息。我们一定要关注其中可能出现的数据。

常见的报错出现在异常处理程序。我们经常需要关注这样几个数据：向通用寄存器（尤其是26-27，也就是上述的k0和k1）中写入的00000010、80000000等。

其实聪明的你一定能够发现，这些通常都是 Cause 寄存器内部的值。联想 Cause 寄存器的功能，能够注意到，在异常判断、延时槽判断等方面出了问题。

更进一步，在课上如果遇到了强测出bug，我们可以编写自己的异常处理代码。首先记得用nop 把测试数据填到4180的地方，第一件事情，就是把 CP0 中寄存器的值都读出来。这里建议大家养成使用宏的好习惯，比如

把中文注释写好，出问题才不会慌乱。
这样，就算是课上出现了问题，冷静下来也能发现问题在哪里。

另外，课上的报错和课下似乎不同（猜的）。课下的报错会明确指出 Wrong interrupt count ，但课上可能不会（目前没听说过），所以，测评机显示未能正确跳转到异常处理程序，也有可能是中断的问题，尤其注意课上的中断测试点。

而关于中断的bug，很有可能来自一下几点

  • HWInt 里，interrupt，TC1_IRQ，TC0_IRQ 的位置错误。（极有可能）
  • 中断信号给出时，M级刚好是一个空泡，即 nop 。
  • 中断信号给出时，M级是一个延时槽指令。

这里只列举了很有可能错的一些点，有很多细节还是要自己慢慢扣的，毕竟P7的细节真的很多。

异常处理程序以外的PC值错误

在课上的强测中，有同学的报错并没有来自于异常处理程序。对于这些bug，我也提出一些比较常见的错误点

  • 乘除槽错误。精确异常要求我们，异常前的指令都要执行完，而异常后的指令都要像从来没有发生过一样。无论是访存还是不涉及乘除槽的计算，产生变化都要在M级或者W级，因此这个要求还算好办。而对于乘除槽则有些麻烦，因为 hi，lo 寄存器的位置在E级，对于 CP0 放在M级的设计，看上去有点难办。
    但仔细想一想，需要关闭乘除槽仅有一种可能，那就是乘除指令紧跟在异常指令之后。所以，处理方法其实很简单。当 CP0 给出 Req 信号的这一周期，关闭乘除槽。若乘除槽内已经有指令正在进行，关闭的这一周期仅仅会使计算延时一周期产生。若异常指令后正好跟着乘除指令，关闭的这一周期会使得该指令无法进入乘除槽，也就杜绝了异常后的乘除指令进入运算的可能。
    在构造相关的测试数据的时候，可以有意的在异常指令周围放一些乘除指令，在异常处理程序里把它从 hi，lo 寄存器中拿出来，对比预期的值是否相同。
  • eret 指令错误。eret 指令不同于其他的跳转指令。作为跳转指令，其没有延时槽（更久远的CPU里需要后接 rfe 指令作为延时槽，这里仅做介绍）。这里建议当D级为 eret 指令时，强行更改F级 PC ，消灭延时槽。另外需要注意的是，eret 指令同时需要清除 SR(EXL) 的置位。

目前想到的可能bug也就这几类，如果有其他想法也欢迎在本帖下方指出~

结语

P7的细节实在是多且繁琐，大家在熬夜写代码的时候也容易手误或者大脑不清醒导致某些地方产生奇妙的bug。随着我们构造的微系统越来越复杂，当输出与预期不符的时候，我们一定要耐下性子观察波形图，从源头上找问题的出现。个人非常建议在观察波形图的时候，从最顶层开始，逐步往内部，顺着电路、顺着流水级“跟踪”一个信号，抑或是追本溯源，从错误地点出发，向前流水级、更上层模块寻找问题产生的地方。不要畏惧波形图的繁琐，这些繁杂错乱的波形图会毫不留情地揭示CPU的错误。

祝愿大家能够顺利完成P7，给自己的计组学习画上完美的句号！
